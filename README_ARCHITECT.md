# Архитектурный паспорт проекта `audio_transcibe_whisper`

## 1. Назначение проекта

Система автоматизирует обработку телефонных разговоров для двух контуров бизнеса:
- `911`
- `КЦ` (контакт-центр, **только кириллический код `КЦ`**)

Проект покрывает полный цикл:
1. загрузка/поиск аудиофайлов,
2. транскрибация (Whisper),
3. хранение артефактов в PostgreSQL,
4. тематическая классификация звонков (Hybrid RAG: Qdrant + Gemma),
5. просмотр, фильтрация и выгрузка через FastAPI + UI.

---

## 2. Бизнес-требования и ограничения

### 2.1 Функциональные требования

1. **Ролевой доступ**:
   - `911` видит только звонки 911,
   - `КЦ` видит только звонки КЦ,
   - `ADMIN` видит всё и управляет справочником/пайплайнами.

2. **Единый справочник тем/подтем** (редактируемый):
   - обязательные поля: тема, подтема, описание, ключевые слова,
   - опциональные: синонимы/фразы, негативные сигналы,
   - хранение в БД + синхронизация в Qdrant.

3. **Автоклассификация звонков**:
   - только по активным транскрипциям,
   - retrieval кандидатов из Qdrant,
   - гибридный скоринг (semantic + lexical + сценарные сигналы),
   - LLM выбирает строго из shortlist кандидатов,
   - результат фиксируется в `call_classifications`.

4. **Поддержка многосоставных звонков**:
   - один звонок может состоять из нескольких частей,
   - агрегируются длительность/время старта и количество частей.

5. **Операционная аналитика**:
   - UI списков звонков и классифицированных звонков,
   - фильтры по периоду/датам/менеджеру/статусу/типу звонка,
   - экспорт в Excel вместе с темой/подтемой и транскрипцией.

### 2.2 Технические ограничения

1. **Канонический код КЦ — `КЦ`** во всех слоях (ингест, API, UI, auth, DB).
2. **Timestamp contract**: таблицы в проекте должны соответствовать `TimestampMixin` (`created_at`, `deleted_at`, `is_active`).
3. **Локальные LLM/Embedding-модели** и внешний Qdrant должны быть доступны по конфигу.
4. **Обязательная проверка консистентности ORM/DB** перед релизом (`db/verify_db_contract.py`).

## 3. Архитектура решения

## 3.1 Компоненты

### A) Data Layer (PostgreSQL + SQLAlchemy)
Основные сущности:
- `users`, `call_types`, `calls`, `call_parts`,
- `transcriptions`, `summarizations`,
- `topic_catalog_entries`, `call_classifications`,
- `pipeline_runs`.

Ключевые инварианты:
- уникальность звонка: `(manager_id, call_type_id, octell_call_id)`,
- уникальность записи справочника: `(topic_name, subtopic_name)`,
- активные версии транскрипции/классификации через `is_active`.

### B) API/UI Layer (FastAPI + Jinja)
Публичные страницы/эндпоинты:
- `/calls`, `/api/calls`, `/api/calls/export.xlsx`,
- `/classified-calls`,
- `/users`, `/api/users`,
- `/catalog`, `/api/catalog` (ADMIN),
- `/pipeline-runs`, `/api/pipeline-runs` (ADMIN).

### C) Classification RAG (КЦ)
Основной production-поток классификации контакт-центра:
1. `classification_rag/sync_topic_catalog.py` — импорт и синхронизация справочника в Qdrant,
2. `classification_rag/classify_calls.py` (или Celery `jobs.classify_calls`) — классификация звонков,
3. `classification_rag/generate_catalog_synonyms.py` (опционально) — помощь в заполнении синонимов.

### D) Summarization LLM (911)
Саммаризация транскриптов 911: код модели и промпта в `summarization_llm/`, постановка задач и запись в БД — `jobs/summarize_911.py` (таблица `summarizations`). Отчёты Excel / Work — отдельные доработки пайплайна.

## 4. Потоки данных (E2E)

## 4.1 Ingest и транскрибация
1. Скрипты ingest (`process_kc_calls.py` / `process_kc_calls_spikers.py`) сканируют папки с аудио.
2. Создаются/обновляются `User`, `CallType`, `Call`, `CallPart`.
3. Whisper создаёт `Transcription` и помечает статус звонка.

## 4.2 Подготовка справочника
1. Бизнес-файл `classification_rag/reference_topics.txt` конвертируется и загружается в `topic_catalog_entries`.
2. Каждая запись синхронизируется как вектор в Qdrant.
3. Для отсутствующих в новом источнике пар тема/подтема запись в БД деактивируется.

## 4.3 Классификация
1. Берутся звонки со статусом `TRANSCRIBED` (или `CLASSIFICATION_FAILED`).
2. Выполняется retrieval в Qdrant.
3. Кандидаты ранжируются гибридным score.
4. Gemma выбирает единственный результат из shortlist или `OTHER`.
5. Результат сохраняется в `call_classifications`, статус звонка обновляется.

## 4.4 Саммаризация (911)
1. После транскрибации для типа звонка 911 выбираются кандидаты (статус `TRANSCRIBED`, нет активной записи в `summarizations` — см. `get_calls_for_summarization`).
2. Celery-задача `jobs.summarize_911_batch` вызывает Gemma через `summarization_llm`, пишет строку в `summarizations`, статус звонка `SUMMARIZED` или `SUMMARIZATION_FAILED`.

## 4.5 Витрины и экспорт
1. UI/API отдают активную транскрипцию и активную классификацию.
2. Excel-экспорт содержит тему/подтему, длительность, части, транскрипцию.

## 5. Что реализовано на текущий момент

1. Полный каркас доменной модели + CRUD для каталога и классификации.
2. Ролевой доступ по API key с разграничением по типам звонков.
3. UI для каталога, классифицированных звонков, пользователей, пайплайнов.
4. Гибридная RAG-классификация с explainability-полями (`evidence`, `reason`, `candidates_json`).
5. Контракт-проверка БД/ORM (`db/verify_db_contract.py`).
6. Smoke-скрипт DB CRUD (`db/test_crud.py`).

## 6. Что доработать в следующей итерации

## 6.1 Приоритет P0
1. Зафиксировать единый migration-runbook и убрать неоднозначность применения миграций для каталога/классификаций.
2. Добавить CI-проверку `verify_db_contract` на каждый merge.
3. Добавить интеграционные тесты API на ключевые сценарии ролей и каталога.

## 6.2 Приоритет P1
1. Версионирование справочника (draft/published) и audit trail изменений.
2. Dashboard качества классификатора:
   - доля `OTHER`,
   - распределение confidence,
   - drift по топ-темам.
3. Улучшить надежность pipeline (retry/backoff на Qdrant/LLM, алерты).

## 6.3 Приоритет P2
1. Bulk-import/bulk-edit справочника через UI.
2. История всех классификаций звонка (не только active).
3. Отдельный аналитический API для BI.

---

## 7. Риски

1. **Схемный дрейф** (ORM и фактическая БД могут расходиться при частично применённых миграциях).
2. **Инфраструктурные зависимости** (Qdrant, локальные модели, GPU/CPU ресурсы).
3. **Качество справочника** напрямую влияет на качество классификации.
4. **Операционные данные** требуют контроля доступа и дисциплины API ключей.

---

## 8. Передача проекта другой команде (handover checklist)

1. Поднять окружение по `README.md` и проверить переменные окружения.
2. Применить согласованный набор миграций.
3. Запустить:
   - `python db/verify_db_contract.py`
   - `python db/test_crud.py`
4. Синхронизировать справочник:
   - `python classification_rag/sync_topic_catalog.py`
5. Выполнить тест-классификацию:
   - `python classification_rag/classify_calls.py --call-type КЦ --limit 20`
6. Проверить UI/экспорт:
   - `/calls`, `/classified-calls`, `/catalog`, `/api/calls/export.xlsx`.

---

## 9. Минимальные критерии приемки релиза

1. `verify_db_contract` проходит без ошибок.
2. Каталог редактируется из UI и запись синхронизируется в Qdrant.
3. Классификация формирует записи в `call_classifications` и меняет статус звонка.
4. Ролевой доступ корректен для 911/КЦ/ADMIN.
5. Excel выгрузка содержит тему/подтему/транскрипцию и не нарушает фильтры роли.
